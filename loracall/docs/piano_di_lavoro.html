LoRaManager slave

Boot sequence + MSG_ANNOUNCE
All'avvio: invia MSG_ANNOUNCE con nome, lista servizi disponibili, flag relay. Riprova ogni 5 secondi finché non riceve MSG_ACK dal master. Dopo 3 tentativi falliti entra in modalità standalone.

ServiceRegistry
Array statico di ServiceDescriptor con service_id, name, active, init_fn, loop_fn, cmd_handler. Funzioni: registerService(), activateService(), dispatchCmd(). Il loop dello slave chiama loop_fn di ogni servizio attivo.

Fase 2 — Dev A Master: console CLI + discovery ~ 2 giorni
Dev A
Console seriale interattiva
Parser comandi su Serial: tokenizza input per spazi, dispatch a handler. Implementare: list (tabella slave), info <id> (dettagli + servizi), ping <id> (RTT), help. Output formattato con colonne fisse.
master/src/console.hmaster/src/console.cpp
Discovery + heartbeat loop
mainLogicTask invia MSG_DISCOVERY_REQ in broadcast ogni DISCOVERY_INTERVAL_MS. Controlla last_seen di ogni slave: se scaduto di HEARTBEAT_TIMEOUT_MS lo marca OFFLINE e stampa avviso in console.
MSG_CMD sender
Comando console "run <id> <svc> <cmd> [args]" costruisce PayloadCmd e lo invia. Attende MSG_CMD_RESULT con timeout CMD_TIMEOUT_MS. Stampa risultato o errore.
Fase 2 — Dev B Slave: primo servizio + CMD handler ~ 2 giorni
Dev B
svc_sensor (DHT22 / ADC)
Implementare il primo servizio concreto. CMD 0x01 = read_all → risponde con JSON minimale {t:22.5,h:60}. CMD 0x02 = set_interval → cambia frequenza di campionamento. Registrarlo nella ServiceRegistry.
slave/services/svc_sensor.hslave/services/svc_sensor.cpp
MSG_CMD dispatcher
Quando arriva MSG_CMD: estrarre service_id e cmd_id, chiamare dispatchCmd() della registry, costruire PayloadCmdResult con esito, inviare MSG_CMD_RESULT al master.
MSG_HEARTBEAT periodico
Task FreeRTOS separato che ogni 10 secondi invia uptime, RSSI dell'ultimo frame ricevuto, heap libero (ESP.getFreeHeap()), bitmask servizi attivi.
Sync punto 2: dal master digitare "run 1 sensor read_all" e vedere la risposta temperatura/umidità in console. Testare "list" con heartbeat aggiornato.
Fase 3 — Dev A OTA sender + Module Composer ~ 3 giorni
Dev A
OTA sender con ARQ
Aprire file dalla SPIFFS/SD, inviare MSG_OTA_START con size+CRC32. Loop: invia chunk → attende MSG_OTA_ACK con chunk_index corrispondente → in caso di timeout ritrasmette fino a ACK_RETRIES. Invia MSG_OTA_END. Stampa progress bar in console (% completato).
master/src/ota_sender.hmaster/src/ota_sender.cpp
Module Composer — config mode
Comando console "compose <id>": mostra lista template disponibili, chiede parametri (pin, SSID, soglie), genera un buffer PayloadConfig con le coppie chiave/valore, lo serializza e lo passa all'OTA sender come OTA_TYPE_MODULE_SEL. Nessuna ricompilazione richiesta.
master/src/composer/module_composer.hmaster/src/composer/module_composer.cpp
Template file sensors + network
Scrivere i file .tpl con placeholder {{PIN}}, {{SSID}}, {{INTERVAL_MS}}. Il code_generator fa una semplice sostituzione stringa. Creare almeno: dht22.tpl, adc_generic.tpl, http_server_simple.tpl.
master/src/composer/templates/
Fase 3 — Dev B OTA receiver + svc_http + svc_relay ~ 3 giorni
Dev B
OTA receiver su partizione ESP32
Gestire MSG_OTA_START: aprire la partizione OTA inattiva con esp_ota_begin(). Per ogni MSG_OTA_CHUNK: esp_ota_write() + inviare ACK_OTA_CHUNK_OK. Su MSG_OTA_END: verificare CRC32, chiamare esp_ota_end() + esp_ota_set_boot_partition() + esp_restart(). In caso di errore CRC inviare ACK_OTA_FAIL.
slave/ota_receiver.hslave/ota_receiver.cpp
Gestione MSG_CONFIG
Ricevere PayloadConfig, iterare i ConfigParam, aggiornare i parametri del servizio target via una funzione setParam(key, value). Persistere in NVS (Preferences.h) in modo che sopravvivano al reboot. Inviare MSG_CONFIG_ACK.
svc_http con interfaccia web
WebServer minimale su porta 80 con una pagina HTML inline che mostra i valori dei sensori attivi aggiornati ogni 5 secondi via fetch(). Attivabile/disattivabile via CMD.
slave/services/svc_http.hslave/services/svc_http.cpp
svc_relay (hop forwarding)
Se attivo, intercetta frame con dst != self_id e hop_count > 0: decrementa hop_count, setta relay_src = self_id, ritrasmette. Non ritrasmettere frame già visti (cache degli ultimi 8 seq recenti per evitare loop).

Sync punto 3: test OTA completo — master invia un file di config allo slave, slave applica, master riceve conferma. Testare relay con 3 nodi fisici.
Fase 4 Integrazione finale, test e hardening ~ 2 giorni
Dev A
Comandi OTA + composer in console
Collegare tutti i pezzi: compose → genera config → send_ota → progress → conferma. Aggiungere comando "set_repeater <id> on|off" e "monitor <id>" per stream output.
Stress test rete
Simulare 4+ slave contemporanei. Verificare che la registry non si corrompa con accessi concorrenti. Testare recovery dopo disconnect/reconnect slave.
Dev B
Robustezza OTA receiver
Testare interruzione OTA a metà trasferimento: il firmware vecchio deve restare attivo (rollback automatico). Testare CRC errato: deve rifiutare e notificare il master senza brickarsi.
Test relay multi-hop
Verificare catena master → repeater → slave finale. Controllare che i frame non vengano duplicati e che hop_count funzioni come TTL.
Dipendenza critica: Dev B non può testare OTA receiver finché Dev A non ha OTA sender funzionante. Pianificare questo sync prima di Fase 3.
